# RAG、微调和评测零基础教程

这份教程面向完全没接触过 RAG、模型微调和召回指标的人。  
你可以把它当成入门地图，不需要先懂所有术语。

## 1. 先搞清楚你在做什么

你现在这个知识库项目，本质上是三层：

1. 数据层：把文档放进来
2. 检索层：把和问题最相关的资料找出来
3. 生成层：让大模型基于这些资料回答

这就是 RAG。

RAG 全称是 Retrieval-Augmented Generation，直译就是“检索增强生成”。

意思很简单：

- 不是让模型凭空记住所有知识
- 而是先查资料，再回答

这样做的好处：

- 企业知识经常更新
- 模型不可能把所有内部资料都背下来
- RAG 比纯微调更灵活，也更适合知识库

## 2. RAG 到底是怎么跑的

一条最小链路通常是：

1. 文档加载
2. 文档切分
3. 文本向量化
4. 建向量索引
5. 用户提问
6. 检索相关片段
7. 把片段塞给模型
8. 模型生成答案

- 文档加载和切分在 [src/kb/ingest.py](../src/kb/ingest.py)
- 问答链在 [src/kb/chain.py](../src/kb/chain.py)
- 评测脚本在 [eval/](../eval/)

## 3. 什么是“召回”

召回讲的是：

> 当用户问一个问题时，你有没有把“正确资料”找出来。

它不是回答对不对，而是“检索对不对”。

举个例子：

- 问题：`企业宽带开通需要多久？`
- 正确资料：`企业宽带常见问题.txt`

如果检索前 5 条里有这份文档，那就是命中。

如果没有，那就是漏召回。

### 常见召回指标

- `Recall@k`
- `Hit@k`
- `MRR`

### 怎么理解

- `Recall@k` 关心“有没有找到”
- `MRR` 关心“排得靠不靠前”

如果检索层都找错了，后面模型再强也没用。

## 4. 什么是 RAG 评测

RAG 评测不是只看答案像不像，而是看两件事：

1. 检索有没有找对资料
2. 模型有没有基于资料回答

常见维度：

- `source_hit_rate`
- `answer_f1`
- `keyword_hit_rate`
- `latency`

你可以把它理解成：

- 检索层：查资料的能力
- 生成层：组织答案的能力

## 5. 什么是模型微调

微调不是“让模型学会查知识库”。

微调更像是：

- 让模型更像你想要的风格
- 更擅长特定格式
- 更懂某类任务

比如：

- 固定输出格式
- 更像客服口吻
- 更会抽取字段
- 更适合分类、摘要、改写

### 微调不适合解决什么

微调通常不适合直接解决：

- 知识频繁更新
- 需要引用具体文档来源
- 需要查最新资料

这些更适合 RAG。

### 一句话区分

- RAG 解决“知识从哪来”
- 微调解决“模型怎么说”

## 6. 你应该先学哪个

建议顺序：

1. 先懂 RAG
2. 再懂评测
3. 再懂微调
4. 最后再看 Agent

## 8. 新手最容易搞混的点

### 1) 召回率高，不代表回答就好

检索找对了，只是第一步。  
如果模型不会组织答案，最后还是会错。

### 2) 回答像真的，不代表有依据

模型可能会编得很像。  
所以要看来源和 groundedness。

### 3) 微调不是万能的

微调不会自动帮你接入最新文档。  
如果知识变化快，RAG 更重要。

### 4) 先做小数据集再谈优化

正确顺序是：

1. 先准备 20-50 条题目
2. 标好正确文档或标准答案
3. 跑基线
4. 再调整 chunk、top-k、prompt、模型

## 9. 一套最小学习路径

### 第一步：先跑通项目

```bash
python -m src.cli build-index
python -m src.cli ask "企业宽带开通需要多久？"
```

### 第二步：看检索结果

```bash
python -m eval.retrieval_eval
```

### 第三步：看回答质量

```bash
python -m eval.rag_eval
```

### 第四步：理解微调能做什么

> 微调是调模型行为，不是替代知识库检索。

## 10. 面试时怎么说

> 我先用 RAG 解决知识更新和可追溯的问题，再用评测数据集去看检索是否命中、回答是否有依据。微调不是第一选择，因为企业知识会频繁变化，RAG 更适合做知识接入；微调更适合做风格、格式、分类、摘要这类稳定任务。
