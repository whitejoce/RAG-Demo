
* 传统RAG ：查询 → 向量化 → 检索 Top-K → 拼 Prompt → LLM 生成
* Agentic RAG  ：查询 → 主代理 → Sub-Agent检索 → 主代理整合 → LLM 生成
``` markdown
用户请求
   ↓
[Orchestrator Agent / 主代理] ← "这个任务需要查资料"
   ↓
[RAG Sub-Agent] ──→ 检索工具 A
      ↓            检索工具 B
   生成草稿       检索工具 C
      ↓
[主代理审核/整合]
   ↓
最终回答
```
可以按四层来测，别混在一起：

**1. 检索层**
目标是看“有没有把对的资料找出来”。

常用指标：
- `Recall@k`：正确片段是否出现在前 `k` 个结果里
- `Hit@k`：有没有命中正确文档
- `MRR`：正确结果排第几，越靠前越好
- `nDCG`：如果你有多级相关性，能看排序质量

怎么做：
- 先做一份小金标集，20-50 个问题就够面试用
- 每个问题标注“应该命中的文档/段落”
- 跑 `top-3 / top-5 / top-8`
- 统计命中率和排名

你这个项目里，检索逻辑主要在 [src/kb/ingest.py](D:/Code/test_codex/src/kb/ingest.py) 和 [src/kb/chain.py](D:/Code/test_codex/src/kb/chain.py)。

**2. RAG 实际表现**
目标是看“找到了资料以后，回答得对不对”。

常用维度：
- `Faithfulness`：回答有没有严格依据检索内容
- `Answer correctness`：答案是否正确
- `Completeness`：有没有漏掉关键点
- `Citation accuracy`：引用是否真来自对应文档
- `Groundedness`：有没有胡编

怎么测：
- 人工打分最靠谱，尤其是小项目
- 每条答案按 1-5 分打分
- 让评审看“问题、检索片段、模型回答、来源”

面试时最重要的一句话是：
- “召回率高，不代表回答就好；RAG 需要同时看检索和生成两层。”

**3. Agent 表现**
目标是看“工具会不会用、任务能不能闭环”。

常用指标：
- `Task success rate`：任务是否完成
- `Tool call accuracy`：该调用工具时有没有调用
- `Tool selection precision`：有没有选对工具
- `Average steps`：完成任务用了几步
- `Retry rate`：失败后重试多少次
- `Cost per task`：token 和工具调用成本

怎么测：
- 准备 10-20 个固定任务
- 每个任务定义成功标准
- 记录 agent 的调用轨迹
- 看是否少走弯路、少重复调用

你现在的 MCP 工具入口在 [src/mcp_server.py](D:/Code/test_codex/src/mcp_server.py)，客户端演示在 [src/mcp_client_demo.py](D:/Code/test_codex/src/mcp_client_demo.py)。

**4. LLM 本身性能**
目标是看“模型本身在这个任务上行不行”。

常测项：
- `准确率`
- `一致性`
- `稳定性`
- `幻觉率`
- `响应延迟`
- `token 消耗`

对你这个场景，最实用的是：
- 固定 prompt
- 固定 temperature = 0
- 固定同一批问题
- 比较不同模型或不同 prompt 的输出

**5. 系统性能**
这个经常被忽略，但面试很加分。

建议测：
- 索引构建时间
- 查询延迟 `p50 / p95`
- 首次冷启动时间
- FAISS 占用内存
- 单次问答 token 数
- API 错误率
- 并发下稳定性

**6. 你还应该了解的内容**
这几个经常被追问：

- 数据集怎么做
  - 先做“金标问答集”
  - 再做“相关文档标注集”
- chunk 怎么调
  - 太小会丢上下文
  - 太大召回噪声高
- top-k 怎么调
  - `k` 小，可能漏召回
  - `k` 大，可能把噪声带进 prompt
- 引用怎么做
  - 最好保留 `source + page + chunk`
- 幻觉怎么压
  - 限制“只能基于资料回答”
  - 检索不到就明确说“不足”
- prompt injection 怎么防
  - 文档内容不能直接覆盖系统提示词
- 评测要可复现
  - 固定模型版本、prompt、温度、检索参数

**7. 对你这个项目最实用的评测方案**
建议你做三张表：

- 检索表：`问题 / 正确文档 / top-k 命中 / Recall@k`
- RAG 表：`问题 / 检索片段 / 模型回答 / 人工评分`
- Agent 表：`任务 / 工具调用次数 / 是否成功 / 总耗时`

这样面试时你能清楚说明：
- 检索层有没有召回
- 生成层有没有答对
- Agent 有没有把工具链跑通
