# 企业政企知识库项目面试学习文档

这份文档讲的是：这个原型是怎么做出来的、核心技术点在哪里、面试时应该怎么讲，以及容易被追问的地方。

## 1. 这个项目解决什么问题

目标很简单：把企业内部的文档资料变成一个可检索、可问答、可接入工具链的知识库。

面试时可以这样概括：

> 我做了一个最小可用的企业知识库原型，流程是文档导入、切分、向量化、向量检索、LLM 生成回答，再把问答能力封装成 MCP Tool，方便后续接入 Agent。

## 2. 整体架构

数据流是这样的：

1. 原始文档放在 `data/raw/`
2. `PyPDFLoader` / `TextLoader` 读取文档
3. `RecursiveCharacterTextSplitter` 把长文切块
4. 智谱 Embedding 把文本转成向量
5. FAISS 在本地保存向量索引
6. 检索到的片段交给 LLM 生成答案
7. `mcp_server.py` 把问答能力暴露成 Tool

你可以把它理解成一个标准 RAG:

- Retrieval: 先找资料
- Augmentation: 把资料拼进提示词
- Generation: 再由模型回答

## 3. 核心实现怎么做的

### 文档加载

代码在 [src/kb/ingest.py](../src/kb/ingest.py)。

- PDF 用 `PyPDFLoader`
- TXT / MD 用 `TextLoader`
- 每个文档都补充统一元数据，比如 `source`、`source_path`、`page`

这么做的原因是：后面回答时可以把来源带出来，方便面试演示和排错。

### 文本切分

还是在 [src/kb/ingest.py](../src/kb/ingest.py)。

- 使用 `RecursiveCharacterTextSplitter`
- `chunk_size=600`，`chunk_overlap=120`
- 分隔符里加入中文标点，避免中文段落被硬切碎

切分的目的：

- 控制每次送给 embedding 和 LLM 的上下文长度
- 提高召回精度
- 降低无关内容混进来的概率

### 向量化

实现见 [src/models/zhipu.py](../src/models/zhipu.py)。

这里没有用 OpenAIEmbeddings，而是封了一层智谱 Embedding 调用：

- 认证方式是 `Bearer API_KEY`
- 接口是 `/paas/v4/embeddings`
- 模型默认 `embedding-2`

面试时可以说：

> 我把 embedding 做成了独立适配层，这样后面想换 OpenAI、智谱或国产模型，只改一处代码。

### 向量存储

用的是 FAISS，本地持久化到 `data/index/faiss/`。

优点：

- 最小可用
- 不依赖外部数据库
- 本地演示快
- 索引文件可以直接复用

这版没有上 Chroma，原因是 MVP 阶段先追求稳定和简单。

### 问答链

实现见 [src/kb/chain.py](../src/kb/chain.py)。

这里用的是新式 LangChain 组合链路：

- `create_retrieval_chain`
- `create_stuff_documents_chain`

为什么不用旧版 `RetrievalQA`：

- 它已经是弃用接口
- 新链路更清晰，后面可扩展性更好
- 面试时说“我用了当前推荐 API”会更稳

回答策略上做了两件事：

- System Prompt 强制“只能依据资料回答”
- 检索结果里保留来源文件名和页码

这能减少模型胡编，也方便解释答案从哪里来的。

### MCP Tool

实现见 [src/mcp_server.py](../src/mcp_server.py)。

这里把问答能力封装成一个 `ask_kb` 工具，目的是让别的 Agent 或客户端程序可以直接调用。

你可以这样讲：

> 我不是只做了一个聊天入口，而是把知识库能力标准化成工具接口，这样后续可以接 Agent 编排、工作流或自动化系统。

## 4. 面试时怎么讲

可以按下面这个顺序说：

1. 背景
2. 架构
3. 核心技术
4. 取舍
5. 可扩展点

一个 30 秒版本：

> 我做了一个企业政企知识库 MVP。先把 PDF 和 TXT 文档加载进来，切块后用智谱 embedding 做向量化，再用 FAISS 做本地向量检索。问答层用 LangChain 的 retrieval chain，把检索结果拼进 prompt 让 LLM 回答。最后我把问答封成 MCP Tool，方便接到 Agent 里做闭环。

一个 2 分钟版本：

> 这个项目主要解决企业文档分散、人工查找慢的问题。我先准备了几份电信政企场景文档，做文档加载和切分，再通过 embedding 和 FAISS 建索引。问答阶段先检索 top-k 相关片段，再把片段和问题一起交给模型生成答案，同时输出来源文件名，保证可解释性。最后我把问答能力暴露成 MCP 工具，方便后续接入 Agent 或其他系统。

## 5. 面试官常问什么

### 1) 为什么要切分文档？

因为长文档直接向量化效果差，而且上下文太长会增加噪声。切分后更利于召回和回答精度。

### 2) 为什么不用 RetrievalQA？

因为它在 LangChain 里已经是弃用接口。新链路 `create_retrieval_chain` 更符合当前生态，也更容易组合其他链路。

### 3) 为什么选 FAISS？

因为这版目标是最小可用原型，FAISS 本地、快、简单，不需要额外服务。

### 4) 为什么输出来源？

知识库项目最怕“答得像真的但其实没依据”。来源是为了让答案可追溯，也方便调试。

### 5) 怎么避免模型幻觉？

这版用了三层控制：

- 只检索相关片段
- Prompt 里限制“只能依据资料回答”
- 让回答带来源，方便人工复核

## 6. 可以追问的技术点

你最好提前准备这些：

- chunk_size 为什么是 600
- chunk_overlap 为什么是 120
- top-k 为什么是 4
- 为什么给 TXT 和 PDF 用不同 loader
- 如何处理文档更新
- 如何做权限控制
- 如何做引用定位到段落或页码
- 如何把 MCP Tool 接到真正的 Agent 里

## 7. 这版项目的边界

这只是演示级原型，不包括：

- 企业真实权限体系
- 增量更新和定时同步
- 多租户隔离
- 复杂重排序
- 高并发在线服务

面试时不要把它说成完整生产系统，应该明确它是“最小可用原型”。

## 8. 你可以直接背的总结

> 这是一个基于 LangChain 的企业知识库 MVP。我把文档解析、切分、embedding、FAISS 检索、LLM 回答和 MCP 工具封装串成一条链。重点不是炫技，而是把一个可演示、可解释、可扩展的 RAG 闭环先跑通。
