# 评测模板

这套评测脚本分四层：

- `retrieval_eval.py`：看检索有没有把对的资料找出来
- `rag_eval.py`：看端到端 RAG 回答质量
- `llm_eval.py`：看纯生成模型在固定上下文上的输出质量
- `agent_eval.py`：看工具调用和任务完成情况

## 数据集

默认使用 `eval/datasets/*.jsonl`，格式是小型人工标注集。  
你可以先用这几份样例跑通，再替换成自己的题库。

## 运行方式

```bash
python -m eval.retrieval_eval
python -m eval.rag_eval
python -m eval.llm_eval
python -m eval.agent_eval
```

如果要测 MCP 路径：

```bash
python -m src.mcp_server
python -m eval.agent_eval --mode mcp
```

## 输出指标

### 检索层

- `hit@k`：前 k 个检索结果中是否包含正确文档的命中率
  > 反映了检索系统“能不能找到匹配信息”的基础能力，通常取 k=1, 3, 5。
- `recall@k`：前 k 个文档包含正确文档数与基准正确文档总数之比
  > 侧重评估对于包含多份相关资料的问题，检索系统找得“全不全”。
- `mrr`：平均倒数排名（Mean Reciprocal Rank），第一个正确检索结果排名的倒数均值
  > MRR 衡量的是“用户第一个有用结果出现在多靠前的位置”的平均水平，是评价问答/检索系统排序质量的关键指标。
- `avg_latency_ms`：平均检索延迟（毫秒）
  > 体现了底层向量库或倒排索引并发检索的执行效率。

### RAG 层

- `source_hit_rate`：引证命中率（生成回答是否成功引用了正确的真实文档）
  > 衡量生成时有无真正利用到检索知识，是判断 RAG 系统是否发生“幻觉”的重要参考。
- `keyword_hit_rate`：关键词命中率（生成回答是否包含标准答案中的核心关键词）
  > 粗略评估生成回答的核心内容覆盖度，看是否命中了关键得分点。
- `avg_answer_f1`：回答的平均 F1 分数，通常基于字词重合度计算回答与标准答案的相似度
  > 综合考量精确率与召回率，反映系统生成的文本与标准参考答案在字面上的吻合程度。
- `avg_latency_ms`：端到端平均响应延迟（毫秒）
  > 囊括了“检索 + 组装提示词 + 大模型生成”的总耗时，直观反映用户侧的等待体验。

### LLM 层

- `avg_answer_f1`：纯文本生成回答与参考答案的平均 F1 分数
  > 在排除检索干扰后，专门评估大模型在给定上下文中阅读理解和提炼总结的能力。
- `keyword_hit_rate`：生成回答中要求覆盖的关键实体或关键词的命中率
  > 考评大模型在生成时会不会遗漏上下文中必须提及的强制信息点。
- `avg_latency_ms`：模型推理平均响应延迟（毫秒）
  > 单纯的大模型 API 或本地推理的耗时成本。

### Agent 层

- `task_success_rate`：最终成功解决并输出正确结果的任务比例
  > Agent 系统的终极北极星指标，代表经历了多轮规划与工具调用后最终“把事做成”的概率。
- `tool_match_rate`：调用工具（步骤、参数）与预期工具链组合匹配的正确率
  > 深入评估模型 Function Calling 的能力：选没选对工具、参数提取填入准不准。
- `avg_latency_ms`：Agent 执行该任务的平均整体耗时（毫秒）
  > 因为 Agent 往往伴随多轮思考与环境交互，这是完成该笔任务的时间总成本。

## 注意

- 这是一套“最小可用评测模板”，适合面试展示和本地迭代。
- 真正严格的效果评测，建议再加人工标注、A/B 对比和更多任务分布。
